## 2026-04-11 HTTP/1.1 固定响应热路径首轮优化

本轮只在 `galay-http` 仓库内修改，依赖继续使用系统安装版 `galay-kernel=3.4.4`、`galay-ssl=1.2.1`。优化范围只覆盖：

- `galay-http/kernel/http/HttpWriter.h`
- `benchmark/B1-HttpServer.cc`
- `benchmark/B14-HttpsServer.cc`

本轮保留的改动：

- `HttpWriter` 新增 `sendView(std::string_view)`，用于发送调用方持有、生命周期受控的连续字节视图，避免重复复制到 writer 内部缓冲区
- `B1-HttpServer` 与 `B14-HttpsServer` 把固定 `200 OK` keep-alive 响应改为静态预构造字节串，并把 `reader` / `writer` 提升为连接级复用对象

fresh 验证命令：

```bash
cmake --build build-ssl --target B1-HttpServer B14-HttpsServer T64-https_writer_steady_state T68-http_writer_send_view -j8
ctest --test-dir build-ssl -R 'T64-https_writer_steady_state|T68-http_writer_send_view' --output-on-failure
ctest --test-dir build-ssl --output-on-failure
```

fresh 结果：

- 定向测试 `2/2` 通过
- `build-ssl` 全量回归 `51/51` 通过

### 同机 A/B 样本

HTTP 基线使用未包含本轮修改的 `c46f733` Release+SSL 构建；该提交与本轮开始前的 `HEAD` 在 `B1/B14/HttpWriter` 三处文件上无差异，因此可作为有效对照。以下样本均为同机、同 `wrk -t4 -c100` 口径。

| Workload | 基线 | 优化后 | 变化 |
| --- | ---: | ---: | ---: |
| `B1-HttpServer` `8s` Round 1 | `249904.67 rps` | `280933.09 rps` | `+12.42%` |
| `B1-HttpServer` `8s` Round 2 | `118495.51 rps` | `128092.42 rps` | `+8.10%` |
| `B14-HttpsServer` `6s` Round 1 | `101835.09 rps` | `104846.13 rps` | `+2.96%` |

结论：

- HTTP 明文固定响应路径收益明确，两轮样本都为正，说明 builder/header/序列化固定开销确实是该 benchmark 的可回收成本
- HTTPS 同口径也转正，但当前增益明显小于明文，说明 TLS 发送与握手外的固定成本占比更高
- 这一轮结果只说明 `B1/B14` benchmark steady-state 被压低了，还不能直接替代 `benchmark/compare/protocols/run_compare.sh` 的 Galay vs Rust 同口径结论；如需确认是否已经对 Rust 追平，仍需后续按协议对比脚本重跑 `http/https`

## 2026-04-06 WSS 双路径优化验收

本轮只在 `galay-http` 仓库内修改，依赖使用系统安装版 `galay-kernel=3.4.4`、`galay-ssl=1.2.1`。构建与验证目录为 `/tmp/galay-http-wss-opt-current`，clean `HEAD` 对照使用 `/tmp/galay-http-head-d2548eb-build`。

约束与方法：

- 计划要求先做 baseline 和 profile，再分别优化 `B7-WssServer` 与 `B8-WssClient`
- macOS `sample` 在当前沙箱内无法附着 benchmark 进程，报错为“try running with sudo”，因此本轮 server/client 热点定位采用同机 clean `HEAD` 基线 + 最小改动 A/B
- 最终保留的改动只有两类：
  - `B8-WssClient` 把 steady-state 统计从全局原子改为协程本地批量汇总，并移除了 steady-state `getMessage(...).timeout(...)`
  - `WsWriter` 为 SSL `sendText/sendBinary` 增加直接编码到复用缓冲区的路径，避免临时 `WsFrame` 组装与额外 payload 拷贝
- 最终回退的试验项：
  - 未实际接入 benchmark 的 `WsSslEchoMachine`
  - `WsReader` split-window SSL 直绑接收窗口试验

fresh 验证命令：

```bash
ctest --test-dir /tmp/galay-http-wss-opt-current --output-on-failure
```

fresh 回归结果：`46/46` 通过。

### 同日 clean HEAD vs 优化后

WSS server 验收口径使用 `B7-WssServer + Go client`；WSS client 验收口径使用 `B8-WssClient -> Rust WSS` 和 `B8-WssClient -> Galay B7`。所有结果均为 `fail=0`。

| Workload | clean `d2548eb` | 优化后 | 变化 |
| --- | ---: | ---: | ---: |
| `B7 + Go` `20/4/256` | `178655.50 rps` | `180635.00 rps` | `+1.11%` |
| `B7 + Go` `60/8/1024` | `184872.62 rps` | `184672.75 rps` | `-0.11%` |
| `B8 -> Rust` `8/4/256` | `168211 msg/s` | `171096 msg/s` | `+1.72%` |
| `B8 -> Rust` `60/8/1024` | `203075 msg/s` | `205526 msg/s` | `+1.21%` |
| `B8 -> Galay B7` `60/8/1024` | `193780 msg/s` | `193604 msg/s` | `-0.09%` |

结论：

- client 侧收益成立，尤其是 `B8 -> Rust` 两条 workload 都稳定转正
- server 侧 `20/4/256` 有小幅提升，但 `60/8/1024` 与 clean `HEAD` 基本持平，不能声称拿到了大幅 server-side 增益
- 这轮双路径优化更准确的结论是：WSS 写侧固定开销被压低了，client benchmark 自身热路径也更干净；但 `B7-WssServer` 在重负载下距离 Rust 对照仍有明显差距，后续需要继续从 runtime / TLS 收发固定成本而不是纯 WebSocket 组帧侧寻找收益

## 2026-04-08 WSS server 持续优化复验

在 `2026-04-06` 双路径验收之后，又继续只在 `galay-http` 仓库内做了三轮 WSS server 定向优化，依赖仍为系统安装版 `galay-kernel=3.4.4`、`galay-ssl=1.2.1`。验证构建目录仍为 `/tmp/galay-http-wss-opt-current`。

本轮实际保留并已提交的增量有三类：

- `edf99cd`：扩展 `WSS` 服务端零拷贝回显命中范围，使 ring buffer wrap 场景也能命中零拷贝
- `0565f9e`：把文本帧 `mask` 与 `UTF-8` 快路径接到 `WebSocketFrame` 已有向量化原语
- `c46f733`：把 `B7-WssServer` steady-state 回显循环改成长生命周期 `Ssl` loop machine，减少每消息一次 awaitable / state machine 构造

### 同机三轮长口径复验

同机、同轮、同 workload 对比口径为：

- `B7-WssServer + Go client`
- `60 connections / 15 seconds / 1024 bytes`

对比对象：

- 当前 `steady-state loop` 版本：`c46f733` 之后继续构建的当前 `HEAD`
- 上一提交：`0565f9e`
- Rust WSS server 对照

| Round | 当前 loop 版本 | 上一提交 `0565f9e` | Rust 对照 |
| --- | ---: | ---: | ---: |
| `1` | `190216.00 rps` | `180506.00 rps` | `186564.07 rps` |
| `2` | `178575.67 rps` | `172502.47 rps` | `176968.87 rps` |
| `3` | `174214.53 rps` | `160303.93 rps` | `154836.87 rps` |

结论：

- `c46f733` 这轮 steady-state loop 优化在三轮长口径里都高于上一提交，收益稳定
- 在这组三轮同机样本中，当前版本也都高于 Rust 对照，因此可以说 `B7-WssServer` 已经在最新同轮样本里追平并转正
- 当前 server 剩余工作重点不再是“是否追平当前样本”，而是如何把结果稳定性继续收敛，避免不同轮次之间的抖动

### WsReader Fast Path 复跑快照

在 `WsWriter` 基线之上，又对 `galay-http/kernel/websocket/WsReader.h` 增加了内部 fast path / 内部批量消费路径：对完整缓冲、普通数据帧直接做前缀扫描、目标消息直拷贝和原地解 mask，其余情况继续回退到原来的 `WsFrameParser::fromIOVec(...)`。完成 `T60-ws_reader_fast_path` 与相关 WebSocket 回归后，重新执行同一组 `ws` / `wss` Rust 对比。

### WsWriter Common Fast Path 复跑快照

随后又在 `galay-http/kernel/websocket/WsWriter.h` 里追加了更窄的 server-side common fast path：仅对 `TcpSocket + use_mask=false + Text/Binary + fin=true` 直接写 header、复用 payload buffer，并保留其余分支继续走原有 fallback。完成 `T61-ws_writer_common_fast_path`、`T47/T55/T60/T61` fresh 回归，以及 reader/writer 诊断钩子清理后，重新执行同一组 `ws` / `wss` Galay vs Rust 对比。

随后又在 `galay-http/kernel/websocket/WsConn.h` 里加入了更窄的 composite echo path，用一次内部 `ReadWrite` state machine 覆盖服务端常见的 `getMessage() -> sendText()/sendBinary()` 往返，并补了 `T62-ws_echo_fewer_wakeups` 与 `T47/T55/T60/T61/T62` fresh 回归。该实现中一度存在 `WsReaderSetting` 生命周期错误，导致 plain `ws` benchmark 下出现 `Message too large` 和大量失败；在把 reader state 改为按值持有 setting 后，重新执行同一组 `ws` / `wss` Galay vs Rust fresh 对比。
