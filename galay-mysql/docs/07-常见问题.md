# 07-常见问题

## 构建与安装

### Q: 最小构建要求是什么？

A: include 路径按当前仓库需要：CMake `>= 3.20`、可用的 C++23 / `std::expected` 工具链、`galay-kernel`、OpenSSL、spdlog。import / module 路径还需要满足 `cmake/option.cmake` 的工具链判定。

### Q: 外部项目应该怎样链接？

A: 安装后使用当前导出的 CMake target：

```cmake
find_package(galay-mysql REQUIRED CONFIG)
target_link_libraries(app PRIVATE galay-mysql::galay-mysql)
```

### Q: 为什么打开了 `GALAY_MYSQL_BUILD_MODULE_EXAMPLES=ON` 还是没有 import 示例？

A: 因为真正决定是否生成 import 示例的是 `GALAY_MYSQL_BUILD_MODULE_EXAMPLES_EFFECTIVE`。如果工具链、生成器或 `clang-scan-deps` 条件不满足，CMake 会自动关闭该路径。

## 异步 awaitable 语义

### Q: 为什么文档里不能写 `auto& aw = client.query(...)`？

A: 因为公开接口返回的是 **Awaitable 值对象**，不是引用。当前头文件签名是按值返回，examples/tests 也都按局部值对象使用。

### Q: 一个 awaitable 需要反复 `co_await` 吗？

A: 当前仓库中不把它当作轮询模型来文档化。真实 examples/tests 的契约是：对每个 awaitable 执行一次 `co_await`，然后检查 `std::expected` 与内层 `optional`。

### Q: 为什么返回类型是 `std::expected<std::optional<T>, MysqlError>`？

A: 这是当前 awaitable `await_resume()` 的真实类型。推荐处理顺序是：

1. 先判断 `!result`
2. 再判断 `!result->has_value()`
3. 最后使用 `result->value()`

### Q: 可以在同一个 `AsyncMysqlClient` 上并发发多个查询吗？

A: 当前仓库没有把这种模式作为推荐用法验证。由于客户端共享 socket、parser 与 ring buffer，建议按单条请求 / 响应流顺序使用一个连接。

## 连接池

### Q: `MysqlConnectionPool::acquire()` 成功后拿到的是什么？

A: 是 `AsyncMysqlClient*`，所有权仍在连接池内部。用完后必须 `pool.release(client)`。

### Q: 忘记 `release()` 会怎样？

A: 池中的可用连接会减少，等待者可能长期拿不到连接。当前源码不会替你自动归还这类裸指针。

## 示例与 target 命名

### Q: 为什么文档里的示例文件名全部是 snake_case？

A: 因为真实文件名就是：

- `examples/include/E1-async_query.cc`
- `examples/include/E2-sync_query.cc`
- `examples/include/E3-async_pool.cc`
- `examples/include/E4-sync_prepared_tx.cc`
- `examples/include/E5-async_pipeline.cc`

对应 target 也直接由这些 basename 派生，例如 `E1-async_query-Include`。

### Q: import 示例 target 名是什么？

A: 真实 target 是：

- `E1-async_query-Import`
- `E2-sync_query-Import`
- `E3-async_pool-Import`
- `E4-sync_prepared_tx-Import`
- `E5-async_pipeline-Import`

只有模块路径可用时这些 target 才会被生成。

## benchmark 与性能数据

### Q: 当前仓库有哪些 benchmark？

A: 只有 `B1-SyncPressure` 与 `B2-AsyncPressure` 两个 target，定义在 `benchmark/CMakeLists.txt`。

### Q: benchmark 里的超时参数名是什么？

A: 当前公共解析器只接受 `--timeout-sec`。

### Q: 为什么性能页没有保留固定 QPS / 延迟表？

A: 因为本次修复没有重新产出可追溯的原始跑分输出。对企业文档来说，没有命令、环境和原始 stdout 的数字不应被当作事实陈述。

## 认证与兼容性

### Q: 当前实现明确覆盖哪些认证插件？

A: 源码显式处理 `mysql_native_password` 与 `caching_sha2_password`，并对未覆盖的认证切换 / 认证分支返回错误。

### Q: 文档为什么不承诺 TLS 传输支持？

A: 因为当前公开 API 与示例没有把 TLS 连接作为已验证能力暴露出来。OpenSSL 依赖在仓库里主要用于认证相关加密辅助。
