# 05-性能测试

本页只保留真实 benchmark 入口、真实命令和当前结果状态。

## 1. Benchmark 真相来源

### B1-EtcdKvBenchmark

| 项目 | 当前值 |
| --- | --- |
| 源文件 | `benchmark/B1-etcd_kv_benchmark.cc` |
| target | `B1-EtcdKvBenchmark` |
| 客户端类型 | 同步 `EtcdClient` |
| 工作模型 | 每线程一个客户端实例 |
| 模式 | `put` 或 `mixed(put+get)` |

### B2-AsyncEtcdKvBenchmark

| 项目 | 当前值 |
| --- | --- |
| 源文件 | `benchmark/B2-async_etcd_kv_benchmark.cc` |
| support | `benchmark/AsyncBenchmarkSupport.cc` |
| target | `B2-AsyncEtcdKvBenchmark` |
| 客户端类型 | 异步 `AsyncEtcdClient` |
| 工作模型 | 每 worker 一个 coroutine + 一个客户端实例 |
| 调度模型 | `Runtime` + `IOScheduler` |
| 模式 | `put` 或 `mixed(put+get)` |

## 2. 命令行参数

`B1-EtcdKvBenchmark` 的参数顺序是：

1. `endpoint`
2. `threads`
3. `ops_per_thread`
4. `value_size`
5. `mode`：`put` 或 `mixed`

推荐命令：

```bash
./build/benchmark/B1-EtcdKvBenchmark http://127.0.0.1:2379 8 500 64 put
./build/benchmark/B1-EtcdKvBenchmark http://127.0.0.1:2379 8 500 64 mixed
```

`B2-AsyncEtcdKvBenchmark` 的参数顺序是：

1. `endpoint`
2. `workers`
3. `ops_per_worker`
4. `value_size`
5. `mode`：`put` 或 `mixed`
6. `io_schedulers`：可选，默认自动推导

推荐命令：

```bash
./build/benchmark/B2-AsyncEtcdKvBenchmark http://127.0.0.1:2379 8 500 64 put
./build/benchmark/B2-AsyncEtcdKvBenchmark http://127.0.0.1:2379 8 500 64 mixed
./build/benchmark/B2-AsyncEtcdKvBenchmark http://127.0.0.1:2379 32 500 64 put 8
```

环境变量：无。

## 3. Benchmark 实际统计项

`B1-EtcdKvBenchmark` 和 `B2-AsyncEtcdKvBenchmark` 当前都会输出：

- `Endpoint`
- `Mode`
- `Workers`
- `Ops/worker`
- `Value size`
- `IO schedulers`：仅 `B2`
- `Total ops`
- `Success`
- `Failure`
- `Duration`
- `Throughput`
- `Latency p50`
- `Latency p95`
- `Latency p99`
- `Latency max`

## 4. 运行前提与注意事项

- 需要一个可写的 etcd v3 HTTP endpoint
- benchmark 会写入 `/galay-etcd/bench/<timestamp>/...`
- `B2` 会写入 `/galay-etcd/bench/async/<timestamp>/...`
- benchmark 目标当前不负责清理所有写入数据
- `mixed` 模式是 `put` 后立即 `get`，不是只读压测
- `B2` 的结果依赖 `workers` 与 `io_schedulers` 组合，不同组合不应混写成同一组性能数字

## 5. 当前结果状态

截至 `2026-03-23`：

- `B1` 已有同步 benchmark 入口
- `B2` 已新增异步 benchmark 入口
- Rust 对照入口现已补齐到 `benchmark/compare/rust/`，并提供统一脚本 `scripts/S2-Bench-Rust-Compare.sh`
- 新数字只有在记录执行日期、完整命令、环境信息和原始输出后，才能写成“当前性能”

因此：

- 旧文档中的吞吐或延迟数字一律视为历史样本
- 它们不是当前版本的“已复算验收结果”

## 6. Rust 对照 benchmark

仓库现在提供与 `B1/B2` 对齐的 Rust 基线：

- `benchmark/compare/rust/src/bin/rust_sync_etcd_bench.rs`
- `benchmark/compare/rust/src/bin/rust_async_etcd_bench.rs`
- `scripts/S2-Bench-Rust-Compare.sh`

Rust 基线选型：

- 同步路径：`reqwest::blocking`
- 异步路径：`reqwest` + `tokio`

统一对比入口：

```bash
GALAY_ETCD_ENDPOINT=http://127.0.0.1:2379 \
GALAY_ETCD_BENCH_WORKERS=4 \
GALAY_ETCD_BENCH_OPS_PER_WORKER=125 \
GALAY_ETCD_BENCH_VALUE_SIZE=32 \
GALAY_ETCD_BENCH_MODE=put \
GALAY_ETCD_BENCH_IO_SCHEDULERS=2 \
./scripts/S2-Bench-Rust-Compare.sh ./build
```

脚本会在两个场景下输出 C++ / Rust 的同参结果：

- `B1-EtcdKvBenchmark` vs `rust_sync_etcd_bench`
- `B2-AsyncEtcdKvBenchmark` vs `rust_async_etcd_bench`

输出摘要会包含：

- `start_time` / `end_time`（UTC）
- `throughput`
- `Latency p50/p95/p99`
- 资源指标仍需人工采集的提示

当前脚本默认值也与这组公开 compare workload 对齐：`workers=4`、`ops_per_worker=125`、`value_size=32`、`mode=put`、`io_schedulers=2`。

## 7. 对外发布规则

- 任何对外发布的 etcd benchmark 结论，都必须附上同机、同构建类型、同 workload 的 Rust 基线结果。
- 发布前必须先跑 `scripts/S2-Bench-Rust-Compare.sh`，并保存该脚本的原始 stdout。
- 如果某个数字没有对应 Rust 基线，只能标注为 `internal-only` 或历史样本，不能作为公开性能结论。

## 8. 重新发布性能数字的最低要求

如果你要把新的 benchmark 结果写回文档，至少同时记录：

- 执行日期
- 完整命令
- etcd 版本
- 机器规格（CPU / 内存 / OS）
- 编译器与构建类型
- 原始输出全文

没有这些元数据时，请不要把数字写成“当前性能”。
