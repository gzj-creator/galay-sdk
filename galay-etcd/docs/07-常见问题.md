# 07-常见问题

本页只回答已经从公开头文件、实现或真实示例中确认过的问题。

## Q1：配置阶段提示缺少依赖，哪些是“真的必需”？

当前 `find_package(... REQUIRED)` 的依赖是：

- `galay-kernel`
- `galay-utils`
- `galay-http`
- `spdlog`
- `PkgConfig` + `simdjson`

其中 `galay-utils` 和 `galay-http` 不是“可选推荐项”，而是当前源码构建必需项。

## Q2：为什么 `127.0.0.1:2379` 会报 endpoint 错？

因为当前 endpoint 解析要求带协议。请使用：

- `http://127.0.0.1:2379`
- `http://127.0.0.1`
- `http://[::1]:2379`

补充说明：

- 端口可以省略；省略时 HTTP 默认 `80`
- `https://...` 虽然能被 endpoint 解析器识别，但当前同步 / 异步客户端都不支持 TLS 连接

## Q3：`request_timeout` 到底管什么？

当前实现里的精确行为是：

- 同步客户端：`request_timeout` 会影响 `connect()` 和后续请求 I/O
- 异步客户端：`request_timeout` 用于 HTTP 请求 awaitable；当前没有公开的“异步 connect 超时”接口

如果你需要统一管理异步建连超时，需要在上层 runtime 或调用侧补控制。

## Q4：`config.keepalive` 和 `keepAliveOnce()` 是一回事吗？

不是。

- `config.keepalive`：传输层 keep-alive，影响 HTTP `Connection` 头；同步客户端还会设置 TCP `SO_KEEPALIVE`
- `keepAliveOnce(lease_id)`：etcd 租约续约 API

所以：

- 打开 `config.keepalive` 不会自动续租
- 使用租约时，仍然要显式调用 `keepAliveOnce()`

## Q5：前缀查询为什么结果和我想的不一样？

因为 `prefix=true` 是字节前缀语义，不是路径语义。

例如：

- `get("/app/config/", true)` 会匹配以 `/app/config/` 开头的键
- `get("/app/config", true)` 会匹配以 `/app/config` 开头的键，范围可能更宽

建议：

- 用统一的 key 命名规范
- 对大前缀查询加 `limit`
- 不要把“是否带尾部斜杠”当成库强制规则

## Q6：Pipeline 到底是什么？会不会“部分失败”？

当前 pipeline 的真实语义是：

- 单次发送到 `POST /v3/kv/txn`
- `compare=[]`
- `success=[ops]`
- `failure=[]`

当前公开 API 没有 compare/failure DSL，因此它不是“完整事务构造器”。

对外可见的失败模型是：

- 参数非法（空 key、非正 limit、非正 lease_id）时，构造请求体阶段直接失败
- HTTP / 服务端 / 解析异常时，整个 `pipeline()` 调用失败
- 成功时，`lastPipelineResults()` 返回与输入顺序一致的逐项结果

因此，不要把当前接口文档理解成“支持任意条件事务”或“暴露部分失败状态机”。

## Q7：可以把同一个客户端实例共享给多个线程或协程吗？

不建议。

原因不是“绝对不能共享”，而是：

- 当前库没有把客户端声明为线程安全
- `lastKeyValues()`、`lastPipelineResults()` 等状态是共享缓存
- 并发操作会互相覆盖最近结果

更稳妥的做法是：

- 同步：每线程一个 `EtcdClient`
- 异步：每协程 / 每业务单元一个 `AsyncEtcdClient`

## Q8：怎么确认服务端到底返回了什么？

失败时请同时记录：

```cpp
result.error().message();
client.lastStatusCode();
client.lastResponseBody();
```

其中：

- `lastStatusCode()` 是最近一次 HTTP 状态码
- `lastResponseBody()` 是最近一次响应体原文

这比只打印 `error().message()` 更适合排查服务器端问题。

## Q9：为什么现在文档里没有“当前 benchmark 数字”？

因为 benchmark 页现在只发布两类结果：

1. 本次修复中真实复算并附带命令 / 环境 / 日期的结果
2. 明确标注为历史样本的旧数据

截至 `2026-03-10`，本次文档修复只重新验证了 benchmark target 的存在与可构建性，没有重新执行性能复测，所以没有发布新的吞吐或延迟数字。

## Q10：为什么示例源码里还看到默认地址？

因为公开 `examples/` / `test/` / `benchmark` 在省略 `argv[1]` 时，会回落到和 `EtcdConfig::endpoint` 一致的本地默认值：`http://127.0.0.1:2379`。

文档的统一建议是：

- 把 endpoint 当作运行参数显式传入
- 企业环境不要依赖源码里的默认本地地址
- 运行命令统一参照 [04-示例代码](04-示例代码.md)
