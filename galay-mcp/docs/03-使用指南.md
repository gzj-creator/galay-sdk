## 选择传输方式

### Stdio vs HTTP

| 维度 | Stdio | HTTP |
| --- | --- | --- |
| 推荐场景 | 本地 CLI、进程内外协作、双向管道调试 | 网络服务、远程访问、协程处理器 |
| 真实示例 | `E1-BasicStdioUsage` | `E2-BasicHttpUsage` |
| 真实测试 | `T1-stdio_client` + `T2-stdio_server` | `T3-http_client` + `T4-http_server` |
| 关键脚本 | `scripts/S2-Run.sh`、`scripts/S4-RunIntegrationTest.sh` | `scripts/S5-TestHttpServer.sh`、`scripts/S7-RunHttpIntegrationTest.sh` |
| 调试特点 | 需要两条 FIFO 保持双向会话 | 必须先 `initialize`，再发 `tools/list` / `tools/call` |

### 选择建议

- 需要最短验证路径时，优先从 `stdio` 套件开始。
- 需要远程接入、HTTP 基础设施或并发压测时，使用 `HTTP` 路径。
- 需要可审计性能数据时，先跑真实 benchmark，再记录运行时间、机器规格和原始输出。

## 推荐工作流

### 1. Stdio 工作流

构建真实入口：

```bash
cmake -S . -B build -DBUILD_MODULE_EXAMPLES=OFF
cmake --build build --target E1-BasicStdioUsage T1-stdio_client T2-stdio_server
```

先跑协议级验证：

```bash
BUILD_DIR=build bash scripts/S2-Run.sh
```

再跑双向集成验证：

```bash
BUILD_DIR=build bash scripts/S4-RunIntegrationTest.sh
```

如果需要手工联调，使用与脚本等价的真实命令：

```bash
mkfifo /tmp/galay-stdio-c2s /tmp/galay-stdio-s2c
./build/bin/T2-stdio_server < /tmp/galay-stdio-c2s > /tmp/galay-stdio-s2c &
SERVER_PID=$!
./build/bin/T1-stdio_client > /tmp/galay-stdio-c2s < /tmp/galay-stdio-s2c
kill "${SERVER_PID}"
rm -f /tmp/galay-stdio-c2s /tmp/galay-stdio-s2c
```

### 2. HTTP 工作流

构建真实入口：

```bash
cmake -S . -B build -DBUILD_MODULE_EXAMPLES=OFF
cmake --build build --target E2-BasicHttpUsage T3-http_client T4-http_server
```

先验证服务端 JSON-RPC 行为：

```bash
./build/bin/T4-http_server 8080 0.0.0.0
HOST=127.0.0.1 PORT=8080 BUILD_DIR=build bash scripts/S5-TestHttpServer.sh
```

再跑真实客户端联调：

```bash
BUILD_DIR=build bash scripts/S7-RunHttpIntegrationTest.sh
```

如果只想看示例程序：

```bash
./build/bin/E2-BasicHttpUsage server
./build/bin/E2-BasicHttpUsage client http://127.0.0.1:8080/mcp
```

当前 `test/T3-http_client.cc` 与 `examples/common/E2-BasicHttpUsageMain.inc` 都使用完成信号等待协程结束，不再依赖固定睡眠时间。

### 3. Benchmark 工作流

构建真实 benchmark：

```bash
cmake -S . -B build -DBUILD_MODULE_EXAMPLES=OFF
cmake --build build --target B1-stdio_performance B2-http_performance B3-concurrent_requests T2-stdio_server T4-http_server
```

典型运行方式：

```bash
./build/bin/B2-http_performance --url http://127.0.0.1:8080/mcp --connections 8 --requests 2000 --io 2 --compute 0
./build/bin/B3-concurrent_requests --url http://127.0.0.1:8080/mcp --workers 10 --requests 100
```

`B1` / `B2` / `B3` 打印的 `Build Timestamp` 来自 `__DATE__` / `__TIME__`，表示**二进制构建时间**，不是本次测量发生时间。实际测试时间请和 benchmark 原始输出一起记录。

## 调试技巧

### 1. 启用日志

当前仓库没有单独的日志开关文档页；最直接的做法是对真实 target 和脚本做 shell 重定向，保留 stdout / stderr：

```bash
./build/bin/T4-http_server 8080 0.0.0.0 >http-server.log 2>&1
BUILD_DIR=build bash scripts/S7-RunHttpIntegrationTest.sh >http-suite.log 2>&1
```

Stdio 路径建议分别保存服务端和客户端输出，便于比对请求 / 响应顺序。

### 2. 查看原始消息

本小节只放**真实命令**；来源分别见 `scripts/S4-RunIntegrationTest.sh`、`test/T1-stdio_client.cc`、`test/T2-stdio_server.cc` 和 `galay-mcp/server/McpHttpServer.cc`。

**Stdio 模式**：

```bash
# stdio 是双向会话，使用两条 FIFO 保存输入输出
mkfifo /tmp/galay-stdio-c2s /tmp/galay-stdio-s2c
./build/bin/T2-stdio_server < /tmp/galay-stdio-c2s > /tmp/galay-stdio-s2c 2>server.log &
SERVER_PID=$!
./build/bin/T1-stdio_client > /tmp/galay-stdio-c2s < /tmp/galay-stdio-s2c | tee client.log
kill "${SERVER_PID}"
rm -f /tmp/galay-stdio-c2s /tmp/galay-stdio-s2c
```

**HTTP 模式**：

```bash
# 新启动的 HTTP 服务端上，先发送 initialize，再发 tools/list
curl -sS -X POST http://127.0.0.1:8080/mcp \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","clientInfo":{"name":"debug-curl","version":"1.0.0"},"capabilities":{}}}'

curl -sS -X POST http://127.0.0.1:8080/mcp \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":2,"method":"tools/list","params":{}}'
```

当前实现里，`handleToolsList()` / `handleToolsCall()` / `handleResourcesList()` 等方法会在连接尚未完成初始化时返回 `Not initialized`；因此首轮原始 HTTP 调试不要跳过 `initialize`。

### 3. 单元测试

当前仓库以**脚本回归测试 + CTest 注册入口**为主，而不是独立的 C++ 单元测试框架。建议直接运行真实回归入口：

```bash
ctest --test-dir build -R galay-mcp-stdio-protocol-suite --output-on-failure
ctest --test-dir build -R galay-mcp-stdio-integration-suite --output-on-failure
ctest --test-dir build -R galay-mcp-http-integration-suite --output-on-failure
```

如果你只验证一个 transport，优先运行对应脚本；如果你要给审计留痕，保留 `ctest` 输出和脚本原始日志。
