# 07-常见问题

## 1. 为什么我只装了 `simdjson` 还是编不过？

因为当前默认库 target `galay-mcp` 会统一编译 HTTP 与 stdio 相关源文件，而 HTTP 公开头文件无条件依赖：

- `galay-http`
- `galay-kernel`

所以当前仓库的默认构建前提实际上是：

- `simdjson`
- `galay-http`
- `galay-kernel`

## 2. 文档里为什么不再推荐 `server | client`？

因为 `stdio` 是双向会话：

- 客户端向 `stdout` 发请求，向 `stdin` 读响应
- 服务端向 `stdout` 写响应，向 `stdin` 读请求

单个 pipe 只有一个方向，不足以承载完整往返。正确做法是双向 FIFO、pty 或类似 transport。

## 3. 我该如何快速验证 `stdio` 服务端？

最小协议验证：

```bash
printf '%s\n' \
  '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","clientInfo":{"name":"faq-check","version":"1.0.0"},"capabilities":{}}}' \
  | ./build/bin/T2-stdio_server
```

完整客户端 / 服务端联调：看 [00-快速开始](00-快速开始.md) 或 [04-示例代码](04-示例代码.md)。

## 4. HTTP 相关 target 为什么有时不生成？

HTTP 示例 / 测试 / benchmark 依赖：

- `galay-http`
- `galay-kernel`
- 默认库 `galay-mcp`

如果外部依赖不完整，CMake 配置或编译阶段会失败；如果示例 / benchmark 开关被关闭，对应 target 也不会生成。

## 5. 如何确认模块示例为什么没生成？

检查根 `CMakeLists.txt` 的模块能力判断，常见原因：

- CMake 版本低于 `3.28`
- 生成器不是 `Ninja` / `Visual Studio`
- 使用 Clang 但找不到 `clang-scan-deps`

## 6. 现在支持 WebSocket 吗？

不支持。仓库当前没有 WebSocket transport 的实现、target 或测试。任何 WebSocket 代码片段都只能作为未来扩展思路，不能当成当前能力。

## 7. 为什么性能页不再给出具体数字？

因为旧页面没有附带与当前提交绑定的原始 benchmark 输出、环境说明和命令证据链。为了避免把不可复核数字当成事实，本次整改把性能页降级为：

- 真实 benchmark target 清单
- 真实命令与参数
- “本次未重新测量”的明确状态说明

## 8. 示例页为什么删掉了一些业务代码片段？

因为它们不是仓库内的真实示例文件，也没有被测试或 benchmark 证明。RAG 场景里，这类片段很容易被误检索为“受支持 API”，所以已从“已验证示例”中剥离。

## 9. 如何找到当前真正可运行的入口？

优先看这些目录：

- `examples/`：演示性质入口
- `test/`：回归测试入口
- `benchmark/`：性能压测入口
- `scripts/`：封装好的运行脚本

再通过 `CMakeLists.txt` 校验 target 名称。
