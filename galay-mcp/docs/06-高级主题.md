# 06-高级主题

本页聚焦当前仓库已经实现、且对企业接入最重要的高级主题：模块构建、HTTP 运行时模型、stdio 双向传输约束，以及扩展边界。

## 1. C++23 模块构建

### 真实条件

根 `CMakeLists.txt` 中的模块能力检查要求：

- CMake `>= 3.28`
- 生成器必须是 `Ninja` 或 `Visual Studio`
- 使用 Clang 时，必须能找到 `clang-scan-deps`

当这些条件不满足时，`BUILD_MODULE_EXAMPLES` 会被自动关闭，并给出禁用原因。

### 相关 target

- `galay-mcp-modules`
- `E1-BasicStdioUsageImport`
- `E2-BasicHttpUsageImport`

### 模块名

```cpp
import galay.mcp;
```

## 2. HTTP 运行时模型

`McpHttpClient` 和 `McpHttpServer` 建立在 `galay-kernel` 协程 / runtime 之上：

- `McpHttpClient` 需要外部传入 `kernel::Runtime&`
- `connect()` / `disconnect()` 返回 awaitable
- `initialize()` / `callTool()` / `listTools()` 等通过 `kernel::Coroutine` 写回 `std::expected<...>&`
- `McpHttpServer` 的工具 / 资源 / 提示处理器也使用协程签名

这意味着：

- HTTP API 与同步 `stdio` API 的编程模型不同
- 文档、示例和业务代码应明确区分“同步 `std::expected`”与“协程结果回填”两种用法

## 3. Stdio 双向传输约束

当前 `McpStdioClient` 与 `McpStdioServer` 都直接绑定：

- 读：`stdin`
- 写：`stdout`

因此：

- 单个 shell pipe 只能提供单向流，不足以支撑完整客户端 / 服务端会话
- 真实联调应使用双向 FIFO、pty、父进程托管的双管道，或自行封装 transport

推荐最小手工联调方式：

```bash
mkfifo /tmp/galay-c2s /tmp/galay-s2c
./build/bin/T2-stdio_server < /tmp/galay-c2s > /tmp/galay-s2c &
./build/bin/T1-stdio_client > /tmp/galay-c2s < /tmp/galay-s2c
```

## 4. 构建图的现实边界

当前仓库的默认库构建存在一个企业接入时需要提前知道的边界：

- `galay-mcp/CMakeLists.txt` 会统一编译 `client/*.cc` 和 `server/*.cc`
- HTTP 头文件直接 `#include "galay-http/..."` 与 `#include "galay-kernel/..."`
- 因而当前没有真正的 “stdio-only 最小依赖构建” 路径

如果未来要支持更细粒度交付，通常需要：

- 拆分 stdio / HTTP 子 target
- 把 HTTP 依赖放到独立组件
- 把 README 与快速开始拆成“stdio-only”和“full stack”两条构建路径

## 5. 已知未实现能力

以下内容当前**不是**仓库现成功能：

- WebSocket 传输
- 文档里未映射到真实文件的 placeholder 业务示例
- 与本次提交绑定的 benchmark 结论页

因此，任何“WebSocket 示例”都只能放在设计讨论中，不能伪装成可运行示例。

## 6. 扩展建议

如果你准备基于当前仓库扩展企业场景，建议优先评估：

1. 是否需要拆分 HTTP / stdio 依赖
2. 是否需要更稳定的 transport 抽象层
3. 是否要把 benchmark 输出纳入 CI 产物
4. 是否要为 docs 增加自动链接 / target / 命令校验
