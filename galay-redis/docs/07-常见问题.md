# 07-常见问题

## 为什么 README 和示例都改成了 `command` / `batch` / `execute`？

因为当前 async 公开头文件、示例和测试都已经迁移到这套接口：

- `RedisCommandBuilder`
- `RedisClient::command(...)`
- `RedisClient::batch(...)`
- `RedisMasterSlaveClient::execute(...)`
- `RedisClusterClient::execute(...)`

旧的 `client.get()` / `client.set()` / `client.pipeline()` 已经不再是当前 async 文档的真实接口。

## `command(...)` 的返回值为什么这么“多层”？

`RedisClient::command(...)` 和 `RedisClient::batch(...)` 在 `co_await` 后返回：

```cpp
std::expected<std::optional<std::vector<RedisValue>>, RedisError>
```

建议这样判断：

```cpp
if (!result) {
    // 错误
} else if (!result.value()) {
    // 没拿到完整回复
} else {
    // result.value().value() 是 std::vector<RedisValue>
}
```

## topology 客户端为什么又不是同一个返回类型？

`RedisMasterSlaveClient::execute(...)` 和 `RedisClusterClient::execute(...)` 的 `co_await` 结果是：

```cpp
std::expected<std::vector<RedisValue>, RedisError>
```

它们不会再包一层 `std::optional`。

## 如何用 `find_package(galay-redis)` 接入？

当前仓库已经导出安装包配置。推荐写法是：

```cmake
find_package(galay-redis CONFIG REQUIRED)
target_link_libraries(your_app PRIVATE galay-redis::galay-redis)
```

如果 `galay-kernel`、`galay-utils` 或 `spdlog` 装在自定义前缀，请把它们的安装目录加入 `CMAKE_PREFIX_PATH`。

## 当前公开 surface 回归应该看哪个文件？

优先看这些文件：

- `test/T15-awaitable_surface.cc`
- `test/T16-rediss_surface.cc`
- `test/T19-rediss_pool_and_topology.cc`

它们分别覆盖 awaitable / pool 精简、`rediss://` facade，以及 TLS pool/topology surface。

优先级建议：

1. `examples/include/E1-async_basic_demo.cc`

## `E3-topology_pubsub_demo` 能证明真实 Sentinel/Cluster 都可用吗？

不能。它主要在单节点上展示 API 形状。真实拓扑能力请看：

- `test/T13-integration_cluster_sentinel.cc`
- `test/integration/run_cluster_sentinel_integration.sh`

## `T13-integration_cluster_sentinel` 为什么会跳过？

测试源码要求：

- `GALAY_IT_ENABLE=1`
- 已有 Cluster + Sentinel 环境

如果你不想手工准备环境，直接运行：

```bash
test/integration/run_cluster_sentinel_integration.sh --build-dir <build-dir>
```

该脚本会：

- 拉起 Docker Compose
- 使用你指定的 `<build-dir>` 构建 `T13-integration_cluster_sentinel`
- 设置所需的 `GALAY_IT_*` 环境变量

如果你的构建目录不是默认的 `build/`，也可以这样传：

```bash
GALAY_BUILD_DIR=build-release test/integration/run_cluster_sentinel_integration.sh
```

## 如何用标准 CTest 跑测试矩阵？

当前 `test/CMakeLists.txt` 已经注册了标准 `add_test(...)` 入口，并按标签分组：

```bash
ctest --test-dir <build-dir> --output-on-failure -L unit
ctest --test-dir <build-dir> --output-on-failure -L redis
GALAY_IT_ENABLE=1 ctest --test-dir <build-dir> --output-on-failure -L integration
```

标签语义：

- `unit`：纯本地离线测试
- `redis`：依赖本地 `127.0.0.1:6379`
- `integration`：真实 Cluster + Sentinel 集成

## `B1-RedisClientBench` 和 `B2-ConnectionPoolBench` 还能用吗？

不能。这两个名字是旧文档中的旧目标名。当前真实 benchmark 目标是：

- `B1-redis_client_bench`
- `B2-connection_pool_bench`

## 为什么模块目标有时不存在？

因为 `galay-redis-modules` 只有在以下条件满足时才会生成：

- CMake `>= 3.28`
- `Ninja` 或 `Visual Studio` 生成器
- `Linux + GCC >= 14`，或
- 非 `AppleClang` 且可找到 `clang-scan-deps`

否则 CMake 会打印原因，并关闭 import compilation。

## `RedisConnectOptions::version` 是 RESP 版本开关吗？

对当前 async 路径来说，不应这样理解。实现里目前把：

- `6` 解释为 IPv6
- 其他值解释为 IPv4

因此企业文档里不要再把它写成“切换 RESP3 的主入口”。

## 为什么 benchmark 页不直接给 QPS 结论？

因为当前仓库并没有随文档提交一份可审计、可复现、与当前 worktree 强绑定的 benchmark 原始结果。现在保留的是：

- 真实目标名
- 真实 CLI
- 输出字段说明
- 方法学建议

如果你要发布性能数字，请运行 benchmark 并保存原始终端输出。
