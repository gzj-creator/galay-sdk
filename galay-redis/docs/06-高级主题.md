# 06-高级主题

本页只收录高级能力、工具链限制与已知权衡，不再混入基础教程、FAQ 或历史性能结论。

## C++23 模块构建

真实模块文件：

- `galay-redis/module/galay.redis.cppm`
- `galay-redis/module/ModulePrelude.hpp`

真实模块目标：

- `galay-redis-modules`

### 当前可用条件

这些条件直接来自 `galay-redis/CMakeLists.txt`：

- CMake `>= 3.28`
- 生成器必须是 `Ninja` 或 `Visual Studio`
- GCC 路径：当前稳定支持仅限 `Linux + GCC >= 14`
- Clang 路径：需要非 `AppleClang`，并且系统里能找到 `clang-scan-deps`

如果条件不满足，CMake 会打印原因，并自动关闭 import compilation。

### 推荐命令

```bash
cmake -S . -B build-mod -G Ninja \
  -DCMAKE_BUILD_TYPE=Release \
  -DBUILD_EXAMPLES=OFF \
  -DBUILD_TESTING=OFF \
  -DBUILD_BENCHMARKS=OFF
cmake --build build-mod --target galay-redis-modules --parallel
```

## 拓扑刷新 single-flight

来源：`test/T12-topology_singleflight.cc`

当前实现里：

- `RedisMasterSlaveClient::refreshFromSentinel()` 会把同一波并发刷新折叠成一次后台任务
- `RedisClusterClient::refreshSlots()` 也做了同样的单飞控制

这意味着：

- 业务层不需要为 burst refresh 再套一层互斥
- 但如果你希望严格控制刷新时机，仍应在上层做节流和观测

## Cluster / Sentinel 的真实边界

### 本地示例边界

`examples/include/E3-topology_pubsub_demo.cc` 会在单一 host/port 上演示：

- `subscribe` / `publish`
- `RedisMasterSlaveClient::execute(...)`
- `RedisClusterClient::execute(...)`

它的价值是 API 形状验证，不是生产级拓扑验证。

### 真实集成边界

真实 MOVED / ASK / Sentinel failover 以这些文件为准：

- `test/T13-integration_cluster_sentinel.cc`
- `test/integration/run_cluster_sentinel_integration.sh`
- `test/integration/docker-compose.cluster-sentinel.yml`

脚本依赖：

- `docker`
- `docker compose`
- `redis-cli`
- 已配置的 CMake 构建目录，或可直接执行的 `T13-integration_cluster_sentinel`

脚本运行方式：

- `test/integration/run_cluster_sentinel_integration.sh --build-dir <build-dir>`
- `GALAY_BUILD_DIR=<build-dir> test/integration/run_cluster_sentinel_integration.sh`
- 如果 `<build-dir>` 省略，脚本会先尝试 `build`，再扫描仓库根目录下已有的 `build*` 目录

这意味着：

- 文档不再假设你只能使用 `build/`
- 如果你把工程配置在 `build-release`、`build-doccheck` 等目录，脚本也能对齐

## 连接池调优关注点

来源：`galay-redis/async/RedisConnectionPool.h`、`benchmark/B2-connection_pool_bench.cc`

优先关注这些配置项：

- 容量：`min_connections`、`max_connections`、`initial_connections`
- 获取/连接/空闲超时：`acquire_timeout`、`connect_timeout`、`idle_timeout`
- 健康检查：`enable_health_check`、`health_check_interval`
- 校验策略：`validate_on_acquire`、`validate_on_return`

调优建议：

- 先固定 `max_connections`，看 `peak_active_connections`
- 再根据 `available_connections` 和 `total_acquired` 调整池大小
- 只有在拿到坏连接证据时，再提高校验强度

## 已知限制与权衡

### 包管理 / 安装

- 当前仓库已经安装导出 `GalayRedisConfig.cmake` / `galay-redisConfig.cmake`
- 安装消费入口是 `find_package(galay-redis CONFIG REQUIRED)` + `galay-redis::galay-redis`
- 如果 `galay-kernel`、`galay-utils` 不在系统默认前缀，需要把它们的安装前缀加入 `CMAKE_PREFIX_PATH`

### async 连接 `version` 字段

- `RedisConnectOptions::version` 仍在头文件里公开
- 当前 async 连接实现中，`version == 6` 被解释为 IPv6；其他值走 IPv4
- 因此它不应被写成“切换 RESP2/RESP3 的用户级主接口”

### `RedisClient` 移动语义

`RedisClient.h` 明确要求：

- 不要在操作进行中移动 `RedisClient`
- 只在所有 awaitable 无效后再移动

### 同步 API 的覆盖度

- `sync/RedisSession.*` 仍保留在源码树中
- 安装规则会排除 `sync/`，模块 `galay.redis` 也不会导出它
- 但当前 `examples/` 与 `benchmark/` 没有同步路径示例
- 如果你要在企业文档中提到它，请明确标记“源码树遗留路径，不属于安装公开合同”

### benchmark 结果

- 当前仓库没有把 benchmark 原始输出固化为可审计产物
- 因此 Markdown 中不再保留未经复现的“权威 QPS 数字”
- 要对外发布数字，请保留本次运行终端输出、环境参数和 commit/worktree 信息

### 平台差异

- AppleClang 当前不支持仓库里的模块依赖扫描路径
- Linux 上如果禁用 `io_uring`，会回退到 `epoll`
- macOS 使用 `kqueue`
- Windows 使用 `IOCP`

这些差异来自 `cmake/option.cmake` 和 `galay-redis/CMakeLists.txt`，不要在文档里再写成单一路径。
