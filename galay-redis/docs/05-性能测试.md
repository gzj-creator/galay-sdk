## 构建前提

- CMake：`3.20+`
- 编译标准：`C++23`
- 构建开关：`BUILD_BENCHMARKS=ON`
- 依赖：`OpenSSL`、`spdlog`、`galay-kernel`、`galay-utils`
- 运行环境：至少一个可访问的 Redis 实例；建议 `Release` 或 `RelWithDebInfo`

```bash
cmake -S . -B build \
  -DCMAKE_BUILD_TYPE=Release \
  -DBUILD_BENCHMARKS=ON \
  -DBUILD_TESTING=OFF \
  -DBUILD_EXAMPLES=OFF
cmake --build build --parallel --target B1-redis_client_bench B2-connection_pool_bench
```

`BUILD_TESTS` 仍可作为兼容别名使用，但基线入口已经统一到 `BUILD_TESTING`。

## `B1-redis_client_bench`

### 当前 mode 与实现路径

- `normal`
  - 每次请求复用预编码 RESP 字节
  - 发送入口是 `RedisClient::commandBorrowed(...)`
- `pipeline`
  - 使用 `RedisCommandBuilder` 聚合编码
  - 发送入口是 `RedisClient::batchBorrowed(builder.encoded(), builder.size())`
- `normal-batch`
  - 使用 `RedisCommandBuilder::append(...)` 生成 `commands()`
  - 发送入口是常规 `RedisClient::batch(...)`

因此当前 `B1` 的 plain `normal` / `pipeline` 结果，应理解为“仓库内 borrowed fast path 的吞吐”；不要把它直接当成一般 owning `command(...)` / `batch(...)` 调用方式的普适基线。

### 当前 CLI

源码中的 usage 为：

```bash
./build/benchmark/B1-redis_client_bench \
  [-h host] [-p port] [-c clients] [-n operations] \
  [-m normal|normal-batch|pipeline] [-b batch_size] \
  [--timeout-ms -1|N] [--buffer-size bytes] [--track-alloc] [-q]
```

### 推荐复现命令

普通模式：

```bash
./build/benchmark/B1-redis_client_bench \
  -h 127.0.0.1 -p 6379 -c 10 -n 1000 -m normal -q
```

pipeline 模式：

```bash
./build/benchmark/B1-redis_client_bench \
  -h 127.0.0.1 -p 6379 -c 10 -n 50000 -m pipeline -b 100 -q
```

normal-batch 模式：

```bash
./build/benchmark/B1-redis_client_bench \
  -h 127.0.0.1 -p 6379 -c 10 -n 50000 -m normal-batch -b 100 -q
```

### 当前输出字段

`B1` 的控制台输出分两段：

- 启动摘要
  - `Host`
  - `Clients`
  - `Operations per client`
  - `Mode`
  - `Batch size`（仅当 mode 为 `pipeline` 或 `normal-batch`）
  - `Timeout (ms)`
  - `Client buffer size`
  - `Allocation tracking`
  - `Planned operations`
    - `pipeline`：`clients * operations`
    - `normal` / `normal-batch`：`clients * operations * 2`
- 结果段
  - `Finished`
  - `Completed clients`
  - `Duration`
  - `Success`
  - `Error`
  - `Timeout`
  - `Ops/sec`（仅当 `Duration > 0`）
  - `Requests measured`（仅当收集到 latency 样本）
  - `Request latency p50 (us)`、`Request latency p99 (us)`（仅当收集到 latency 样本）
  - `Success rate`（仅当 `Success + Error + Timeout > 0`）
  - `Alloc calls/op`、`Alloc bytes/op`（仅当 `--track-alloc` 且总样本数大于 0）

```bash
  -h 127.0.0.1 -p 6379 -c 20 -n 300 -m 4 -x 20 -q
```

### 当前输出字段

`B2` 的控制台输出也分两段：

- 启动摘要
  - `Host`
  - `Workers`
  - `Operations per worker`
  - `Pool min/max`
  - `Planned operations`（固定按 `workers * operations * 2` 打印）
- 结果段
  - `Finished`
  - `Duration`
  - `Success`
  - `Error`
  - `Timeout`
  - `Ops/sec`（仅当 `Duration > 0`）
  - `Success rate`（仅当 `Success + Error + Timeout > 0`）
  - `Pool stats`
    - `total_connections`
    - `available_connections`
    - `active_connections`
    - `total_acquired`
    - `total_released`
    - `peak_active_connections`

## `B3-rediss_client_bench`

### 当前 CLI

源码中的 usage 为：

```bash
./build/benchmark/B3-rediss_client_bench \
  [--url rediss://host:port/db] [-c clients] [-n operations] \
  [-m normal|pipeline] [-b batch_size] [--timeout-ms N] \
  [--buffer-size bytes] [--ca-cert path] [--verify-peer 0|1] [--server-name host] [-q]
```

### 推荐复现命令

```bash
./build/benchmark/B3-rediss_client_bench \
  --url rediss://localhost:16380/0 \
  -c 10 -n 500 -m normal -q \
  --verify-peer 0
```

如果你启用证书校验，请额外传 `--ca-cert` 和 `--server-name`。

## benchmark caveats

- `--track-alloc` 会改变开销，不适合与未开启统计的结果混合比较
- `B1` 的 plain `normal` / `pipeline` 当前走 borrowed fast path；如果你要和其他语言/客户端做公平对齐，请先确认双方比较的是同一类调用路径
- `B1` 的 `pipeline` 与 `normal-batch` 代表两种不同工作负载，不能直接按同一基数解读
- `B2` 统计的是连接池 worker 层吞吐，不是单条 Redis 命令的纯网络极限
- 当前仓库没有自动把 benchmark 输出写回 Markdown；你需要自己保存原始终端输出
