# 03-使用指南

本页只给出当前公开头文件、真实示例、真实测试里可落地的工作流。每个条目都标注来源文件、target、运行命令、环境变量与执行模式，避免再出现“代码片段能看不能跑”的漂移。

## 通用前提

- 编译器：支持 C++23
- CMake：`>= 3.16`
- 头文件依赖：可解析 `<concurrentqueue/moodycamel/*.h>`
- 默认开发构建：

```bash
cmake -S . -B build -DBUILD_TESTING=ON -DBUILD_EXAMPLES=ON -DBUILD_BENCHMARKS=ON
cmake --build build --parallel
```

## 1. 启动 Runtime 并运行 Task

- 真实头文件：`galay-kernel/kernel/Runtime.h`
- 真实验证：`test/T22-runtime_stress.cc`
- 真实验证：`test/T37-runtime_strict_scheduler_counts.cc`
- target：`T22-runtime_stress`、`T37-runtime_strict_scheduler_counts`
- 运行命令：

```bash
cmake --build build --target T22-runtime_stress T37-runtime_strict_scheduler_counts
./build/bin/T22-runtime_stress
./build/bin/T37-runtime_strict_scheduler_counts
```

- 环境变量：无
- 执行模式：自测型测试 target

最小用法：

```cpp
Runtime runtime = RuntimeBuilder()
    .ioSchedulerCount(0)
    .computeSchedulerCount(1)
    .build();

Task<int> answer() {
    co_return 42;
}

int value = runtime.blockOn(answer());

auto handle = runtime.spawn(answer());
handle.wait();
int joined = handle.join();
```

注意：

- `Runtime::blockOn(Task<T>)` 会在需要时启动 runtime，并同步返回结果或重抛异常
- `Runtime::spawn(Task<T>)` 返回 `JoinHandle<T>`；公开结果路径统一为 `join()` / `wait()`，drop handle 默认表示 detach
- `Runtime::spawnBlocking(...)` 与 `RuntimeHandle::spawnBlocking(...)` 返回同一套 `JoinHandle<T>` 语义
- `Runtime::handle()` 可把当前 runtime 绑定成轻量 handle，适合在 task 内继续派生任务
- `RuntimeHandle::current()` / `tryCurrent()` 只在 runtime 上下文内可用
- `0` 表示禁用对应默认调度器；自动数量请使用 `GALAY_RUNTIME_SCHEDULER_COUNT_AUTO`

## 10. 构建协议解析型组合 Awaitable

- 真实头文件：`galay-kernel/kernel/Awaitable.h`
- 真实头文件：`galay-kernel/common/ByteQueueView.h`
- 真实验证：`test/T56-custom_sequence_awaitable.cc`
- 真实验证：`test/T59-sequence_parser_need_more.cc`
- 真实验证：`test/T60-sequence_parser_coalesced_frames.cc`
- 运行命令：

```bash
cmake --build build --target \
  T56-custom_sequence_awaitable \
  T59-sequence_parser_need_more \
  T60-sequence_parser_coalesced_frames
./build/bin/T56-custom_sequence_awaitable
./build/bin/T59-sequence_parser_need_more
./build/bin/T60-sequence_parser_coalesced_frames
```

- 环境变量：无
- 执行模式：测试驱动示例

Builder 线性组合：

```cpp
using Result = std::expected<size_t, IOError>;

struct Flow {
    void onSend(SequenceOps<Result, 4>&, SendIOContext& send_ctx) {
        send_ok = send_ctx.m_result.has_value();
    }

    void onRecv(SequenceOps<Result, 4>& ops, RecvIOContext& recv_ctx) {
        ops.complete(std::move(recv_ctx.m_result));
    }

    bool send_ok = false;
};

auto sequence = AwaitableBuilder<Result, 4, Flow>(&controller, flow)
    .send<&Flow::onSend>(payload, payload_len)
    .recv<&Flow::onRecv>(buffer, sizeof(buffer))
    .build();
```

Builder 多段 IO：

```cpp
using Result = std::expected<size_t, IOError>;

struct ScatterFlow {
    void onWritev(SequenceOps<Result, 4>& ops, WritevIOContext& write_ctx) {
        if (!write_ctx.m_result) {
            ops.complete(std::unexpected(write_ctx.m_result.error()));
            return;
        }
    }

    void onReadv(SequenceOps<Result, 4>& ops, ReadvIOContext& read_ctx) {
        ops.complete(std::move(read_ctx.m_result));
    }
};

std::array<struct iovec, 2> out_iovecs{};
std::array<struct iovec, 2> in_iovecs{};
ScatterFlow flow;

auto awaitable = AwaitableBuilder<Result, 4, ScatterFlow>(&controller, flow)
    .writev<&ScatterFlow::onWritev>(out_iovecs, out_iovecs.size())
    .readv<&ScatterFlow::onReadv>(in_iovecs, in_iovecs.size())
    .build();
```

协议解析组合：

```cpp
using Result = std::expected<std::string, IOError>;

struct Flow {
    void onRecv(SequenceOps<Result, 4>& ops, RecvIOContext& recv_ctx) {
        if (!recv_ctx.m_result) {
            ops.complete(std::unexpected(recv_ctx.m_result.error()));
            return;
        }
        inbox.append(scratch, recv_ctx.m_result.value());
    }

    ParseStatus onParse(SequenceOps<Result, 4>& ops) {
        if (!inbox.has(sizeof(uint32_t))) {
            return ParseStatus::kNeedMore;
        }
        const uint32_t body_len = decodeHeader(inbox.view(0, sizeof(uint32_t)));
        if (!inbox.has(sizeof(uint32_t) + body_len)) {
            return ParseStatus::kNeedMore;
        }
        ops.complete(std::string(inbox.view(sizeof(uint32_t), body_len)));
        inbox.consume(sizeof(uint32_t) + body_len);
        return ParseStatus::kCompleted;
    }

    ByteQueueView inbox;
    char scratch[4096]{};
};

auto sequence = AwaitableBuilder<Result, 4, Flow>(&controller, flow)
    .recv<&Flow::onRecv>(flow.scratch, sizeof(flow.scratch))
    .parse<&Flow::onParse>()
    .build();
```

注意：

- 直接使用 `IOController + AwaitableBuilder` 时，底层 fd 需要先处于 non-blocking；高层 `TcpSocket` 仍建议通过 `option().handleNonBlock()` 统一处理
- `parse()` 只能建立在最近一个非本地 IO 步骤之后；典型模式是 `recv(...).parse(...)`
- `ParseStatus::kNeedMore` 不会恢复协程，而是自动重挂最近一个读步骤；这个读步骤可以是 `recv(...)`，也可以是 `readv(...)`
- `ParseStatus::kContinue` 会继续本地 parse loop，适合一次 `recv` 中解析多帧
- `ByteQueueView` 负责保留半包和消费已解析字节；不要在 parse handler 里直接依赖一次 `recv` 的临时缓冲生命周期
- machine-backed builder handler 不支持 `ops.queue(...)`；如果需要手工排队步骤，请切回显式 `SequenceAwaitable + SequenceStep`

状态机入口：

```cpp
using Result = std::expected<std::string, IOError>;

struct Machine {
    using result_type = Result;

    MachineAction<result_type> advance();
    void onRead(std::expected<size_t, IOError> result);
    void onWrite(std::expected<size_t, IOError> result);
    // 只有使用 MachineSignal::kWaitConnect 时才需要：
    void onConnect(std::expected<void, IOError> result);
};

auto awaitable = AwaitableBuilder<Result>::fromStateMachine(
    &controller,
    Machine{}
).build();
```

状态机动作约束：

- `MachineSignal::kContinue`：继续本地推进，适合 parser backlog 或纯状态切换
- `MachineSignal::kWaitRead / kWaitReadv / kWaitWrite / kWaitWritev / kWaitConnect`：让内核注册下一次读写或连接完成等待
- `MachineSignal::kComplete`：用 `result_type` 收口最终结果
- `MachineSignal::kFail`：用于状态机契约错误或内核级失败；领域错误应放进 `result_type`
- 如果状态机会返回 `MachineSignal::kWaitConnect`，需要额外实现 `onConnect(std::expected<void, IOError>)`

当前实现说明：

- 链式 `AwaitableBuilder` 的 `build()` 已统一落到共享状态机内核
- 线性多段收发可以直接用 `readv(...) / writev(...)`，不需要再额外写专用 Awaitable
- 常见 `connect / recv / parse / send / local / finish` 语义保持不变
- 显式 `SequenceAwaitable + SequenceStep` 仍然可用，适合需要手工编排步骤实例的场景

如何选择：状态机还是 Builder

- 先看 `examples/include/E10-custom_awaitable.cc`：
  适合 `connect / read / write / shutdown / handshake` 这类状态切换多、推进路径不线性的场景。
- 先看 `examples/include/E11-builder_protocol.cc`：
  适合 `recv -> parse -> send` 这类线性协议流，尤其是需要 `ByteQueueView + ParseStatus::kNeedMore` 处理半包时。
- 如果你的 handler 需要显式 `ops.queue(...)`，或者必须长期持有步骤对象，请不要混用 machine-backed builder，改用显式 `SequenceAwaitable + SequenceStep`。

最小入口对照：

```cpp
// 状态机：自定义 Awaitable 骨架
auto awaitable = AwaitableBuilder<Result>::fromStateMachine(
    &controller,
    Machine{}
).build();

// Builder：常见协议骨架
auto awaitable = AwaitableBuilder<Result, 4, Flow>(&controller, flow)
    .recv<&Flow::onRecv>(flow.scratch, sizeof(flow.scratch))
    .parse<&Flow::onParse>()
    .send<&Flow::onSend>(reply.data(), reply.size())
    .build();
```

经验上可以这样判断：

- 业务逻辑天然是“一条线”：优先 Builder
- 业务逻辑天然是“多状态推进”：优先状态机
- 你想先学最底层扩展模型：先看 `E10`
- 你想先写最常见的协议组合：先看 `E11`
