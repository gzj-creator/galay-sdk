# 06-高级主题

## 1. 后端选择与平台行为

后端由 `cmake/option.cmake` 自动判断：

- Linux + `DISABLE_IOURING=ON`：epoll
- Linux + `DISABLE_IOURING=OFF` 且找到 `liburing`：io_uring
- macOS：kqueue
- Windows：planned，当前配置阶段直接报错，不生成可用后端

这会影响：

- `USE_EPOLL` / `USE_IOURING` / `USE_KQUEUE` 宏
- 某些 API 的可用性，例如 `AioFile` 只在 `USE_EPOLL` 下公开

## 2. C++23 命名模块

源码门面：`galay-kernel/module/galay.kernel.cppm`

真实条件：

- `ENABLE_CPP23_MODULES=ON`
- `CMake >= 3.28`
- 生成器支持模块
- 编译器支持该项目的模块配置

当前环境验证结果：

```bash
cmake -S . -B build-docfix-modules -DENABLE_CPP23_MODULES=ON
```

输出表明：

- `Unix Makefiles` 会让 CMake 自动关闭模块
- `AppleClang` 也会让该项目自动关闭模块

因此当前环境不会生成 `galay-kernel-modules` 与 import 示例 target。

## 3. Runtime 绑核

`RuntimeBuilder` 提供两种高级绑定方式：

- `sequentialAffinity(io_count, compute_count)`：从 CPU 0 起顺序分配
- `customAffinity(io_cpus, compute_cpus)`：显式指定 CPU 列表

约束：

- 自定义列表长度必须与调度器数量严格一致
- 否则 `customAffinity(...)` 返回 `false`

## 5. 文件监控

`FileWatcher` 的真实模型是：

- 构造：`FileWatcher()`
- 注册：`addWatch(path, events)`
- 等待：`co_await watch()`

不是“构造时传入 scheduler”。

## 6. 组合 Awaitable 与协议解析

`v3.2.0` 起，低层组合扩展统一收口到：

- `SequenceAwaitable<ResultT, InlineN>`
- `SequenceStep<ResultT, InlineN, FlowT, BaseContextT, Handler>`
- `AwaitableBuilder<ResultT, InlineN, FlowT>`
- `StateMachineAwaitable<MachineT>`
- `MachineAction<ResultT>`
- `ParseStatus`
- `ByteQueueView`

`v3.3.0` 在这套扩展面上继续补充：

- `AwaitContext`：在挂起前把当前 `TaskRef` 与 `Scheduler*` 注入给 state machine 或 builder flow
- `SequenceOwnerDomain`：把 sequence owner 显式建模成 `Read / Write / ReadWrite`
- 共享 state-machine `.timeout(...)`：direct / builder 路径统一返回超时错误

推荐分工：

- 线性 IO pipeline / header-body 分段收发：优先 `AwaitableBuilder`，多段缓冲直接用 `readv/writev`
- 复杂读写切换 / handshake / shutdown：优先 `AwaitableBuilder::fromStateMachine(...)`
- 需要显式步骤实例或更细粒度复用：使用 `SequenceAwaitable + SequenceStep`
- 协议解析：`recv(...).parse(...) + ByteQueueView`

共享内核：

- 链式 `AwaitableBuilder` 的 `build()` 现在不再直接拥有独立的 sequence 执行内核
- 线性 builder 会先编译为内部 `LinearMachine`
- `fromStateMachine(...)` 与链式 builder 最终都复用同一个 `StateMachineAwaitable` 驱动

双工 owner 规则：

- 同一 `IOController` 现在允许一个只读 sequence owner 与一个只写 sequence owner 并发共存
- 双向 `StateMachineAwaitable` 默认仍按 `ReadWrite` 独占双向 owner
- 普通 `recv/send` awaitable 与同方向 sequence owner 仍保持 fail-fast 互斥，避免底层 fd 字节流归属混乱

状态机动作模型：

- `MachineSignal::kContinue`：继续本地推进
- `MachineSignal::kWaitRead`：挂读事件
- `MachineSignal::kWaitReadv`：挂多段读事件
- `MachineSignal::kWaitWrite`：挂写事件
- `MachineSignal::kWaitWritev`：挂多段写事件
- `MachineSignal::kWaitConnect`：挂连接完成事件
- `MachineSignal::kComplete`：完成并唤醒协程
- `MachineSignal::kFail`：以内核级 `IOError` 收口

解析语义：

- `ParseStatus::kNeedMore`：不恢复协程，自动重挂最近一个非本地 IO 步骤；读步骤可以是 `recv/readv`
- `ParseStatus::kContinue`：不等待新的内核事件，继续本地 parse loop
- `ParseStatus::kCompleted`：当前逻辑帧已收口，可以完成结果或进入下一阶段

这套语义专门解决：

- TCP 半包：头不完整、body 不完整时不提前恢复上层协程
- TCP 粘包：单次 `recv` 带来多帧时，尽量在一次逻辑恢复里吃完 backlog
- 生命周期：业务库不需要再继承旧 `CustomAwaitable` 自己管理一串步骤对象
- 显式步骤排队：如果确实要在 handler 里 `ops.queue(...)`，请直接使用 `SequenceAwaitable + SequenceStep`，不要混用 machine-backed builder
